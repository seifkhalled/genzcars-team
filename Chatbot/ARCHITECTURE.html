Deals Chatbot — Multi-Agent Architecture v2.0.0

LangGraph-powered multi-agent chatbot with hybrid search, MCP tool calling, type-safe enums, and LangSmith observability.

1. Service Topology

Chatbot

FastAPI :8001

The multi-agent brain. Hosts the LangGraph graph, all 10 agent nodes, LLM routing (Groq + OpenRouter), embeddings (FastEmbed), hybrid search, SSE streaming, and MCP tool servers.

Backend

FastAPI :8000

Car ads CRUD, auth, and SSE proxy (/message → Chatbot). Manages Supabase PostgreSQL for ads, favorites, and users.

Comparison

FastAPI :8002

Dedicated analysis pipeline: structured comparison, vision analysis, hallucination guard, and report generation.

Frontend

Next.js

Next.js SSR app with bilingual (AR/EN) support. SSE consumer for chatbot streaming responses.

Qdrant

Vector DB

Cloud-hosted vector DB (cars_ads). Hybrid index: BM25 + dense vector. Stores car ad embeddings.

Supabase

PostgreSQL

Persistent storage: chat_messages, user_preferences, chat_sessions, ads.

2. Type-Safe Enums

All node names and task types use str, Enum from app/enums.py, replacing bare string literals across the codebase. This prevents silent routing bugs from typos and enables IDE autocompletion.

NodeName 10 members

Used in builder.py, router.py, constants.py, responder_node.py.

class NodeName(str, Enum):
    PREFERENCE_EXTRACTOR = "preference_extractor"
    ROUTER               = "router"
    CATALOGUE            = "catalogue_node"
    SEARCH               = "search_node"
    RECOMMENDATION       = "recommendation_node"
    ADVISOR              = "advisor_node"
    SELLER               = "seller_node"
    GUIDE                = "guide_node"
    GENERAL              = "general_node"
    RESPONDER            = "responder_node"

TaskType 11 members

Used in core/llm.py for SIMPLE_TASKS / COMPLEX_TASKS classification, and across all 7+ node files for ainvoke_task() / astream_task() calls.

class TaskType(str, Enum):
    ROUTER               = "router"
    PREFERENCE_EXTRACTOR = "preference_extractor"
    SEARCH               = "search"
    CATALOGUE_CHECK      = "catalogue_check"
    GUIDE_TOPIC          = "guide_topic"
    SEARCH_DECISION      = "search_decision"
    ADVISOR              = "advisor"
    SELLER               = "seller"
    RECOMMENDATION       = "recommendation"
    GENERAL              = "general"
    COMPARISON           = "comparison"
StrEnum behavior: Both enums extend str, Enum, so NodeName.CATALOGUE == "catalogue_node", TaskType.SEARCH in SIMPLE_TASKS, and {"catalogue_node": x}.get(NodeName.CATALOGUE) all work naturally. No .value calls needed anywhere.

SELLER nodes/seller_node.py messages[-1], preferences node_response, price_analysis 1-2x Groq (SELLER)
GUIDE nodes/guide_node.py messages[-1] node_response 2x Groq (GUIDE_TOPIC, GUIDE)
GENERAL nodes/general_node.py messages[-1], preferences node_response, preferences 2-3x Groq (SEARCH_DECISION, GENERAL, PREFERENCE_EXTRACTOR)
RESPONDER nodes/responder_node.py node_response, retrieved_ads messages (AIMessage) None (pure SSE emission)
Key design principle: Every node receives RunnableConfig so dependencies (LLMs, Qdrant, SSE queue, DB pool) are injected at runtime — no global singletons inside nodes.

5. Tool Calling (MCP)

car-search Server

Tools:

  • search_cars — hybrid (BM25+vector) search with metadata filters
  • get_car_details — full ad from PostgreSQL by UUID
  • find_similar_cars — similar by brand/model/year
  • check_catalogue — availability check for specific make/model
  • get_car_images — image URLs for ad IDs

Dependencies: QdrantSearch, Embedder, DB pool

analysis Server

Tools:

  • analyze_market_price — median/range from similar listings
  • web_search — DuckDuckGo web search for market context
  • get_website_guide — website feature reference for guide_node
  • expand_brand_origins — "German car" → [BMW, Mercedes, VW]
  • expand_colloquial_terms — slang → standard vocabulary

Dependencies: QdrantSearch, Embedder, WebSearch

6. Memory & Context Management

Graph State transient

CarsChatState holds: messages (add_messages reducer), preferences, retrieved_ads, node_response, intent, turn_count, intent_history. Reset per turn; previous turns loaded from DB.

Supabase persistent

chat_messages: role, content, node_used, ad refs. user_preferences: all preference fields + intent_history + turn_count. Loaded at turn start, saved fire-and-forget.

MemorySaver checkpointer

LangGraph's MemorySaver() for intra-stream state persistence. Not used for cross-turn persistence.

Semantic Cache in-memory

embedding_cache (1000 entries, 2h TTL) and llm_response_cache (300 entries, 30min TTL). LRU eviction. Cacheable task types: ROUTER, GUIDE_TOPIC, SEARCH_DECISION.

Intent History persistent

intent_history tracks which NodeName handled each turn (e.g. ["catalogue_node", "advisor_node"]). Stored in user_preferences.

CostTracker per-request

BaseCallbackHandler per turn. Tracks tokens, cost, latency per LLM call. Per-node tag via set_node_tag().

7. LLM Routing & Fallback

MultiLLM Strategy

CategoryTaskType MembersPrimary ModelFallback Chain
SIMPLE_TASKS ROUTER, PREFERENCE_EXTRACTOR, GUIDE_TOPIC, SEARCH_DECISION, CATALOGUE_CHECK Groq (fast): openai/gpt-oss-120b, temp=0 fast model only
COMPLEX_TASKS ADVISOR, SELLER, SEARCH, RECOMMENDATION, GENERAL, COMPARISON Groq (powerful): qwen/qwen3.6-27b, temp=0.3 Groq powerful → OpenRouter → Groq fast 
MultiLLM.ainvoke_task(task_type, messages)
│
├─ task_type in SIMPLE_TASKS? → Groq fast (no fallback, timeout 25s)
│
├─ task_type in COMPLEX_TASKS? → try in order:
│     1. Groq powerful    (fallback model, streaming=True)
│     2. OpenRouter       (nemotron-3-nano, fallback API key)
│     3. Groq fast        (last resort)
│
├─ Cache hit? (cacheable: ROUTER, GUIDE_TOPIC, SEARCH_DECISION)
│     → return cached LLMResult immediately
│
└─ All failed? → raise RuntimeError("Service temporarily unavailable")
