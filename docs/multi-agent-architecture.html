2. Node Details & Tool Calling

preference_extractor every turn
Extract and merge all car-preference signals from the user's latest message into accumulated UserPreferences. Runs on every turn before routing.
LLM 1× FallbackLLM.ainvoke() — JSON preference extraction
Tool Calls
  • format_brand_origins_prompt() — expand country/region mentions to brand lists
  • upsert_user_preferences() — persist to PostgreSQL (fire-and-forget)
→ router (fixed)
router every turn
Intent classifier — decides which specialist node should handle the request based on conversation context, preferences, and latest message.
LLM 1× FallbackLLM.ainvoke() — classify into one of 5 intents
Tool Calls
  • None (pure LLM classification)
→ catalogue_node | advisor_node | seller_node | guide_node | general_node (conditional via next_node state)
catalogue_node
Determine if the request is for a specific known car or broad search. If specific, checks database catalogue availability.
LLM 1× FallbackLLM.ainvoke() — extract specific vs broad request
Tool Calls
  • format_brand_origins_prompt() — brand origin expansion
  • check_catalogue_availability() — SQL query to count exact-match ads
→ search_node (available or broad) | recommendation_node (not available)
search_node
Vector-based semantic search against Qdrant. Builds query, enriches with preference filters, evaluates quality, potentially retries broadly.
LLM 1× ainvoke (query build) + 1× astream (response gen)
Tool Calls
  • format_expansions_prompt() — expand colloquial car terms
  • embedder.encode() — text → vector embedding
  • qdrant_search.search() — vector search with metadata filters
  • verify_results() — filter to active listings only
  • evaluate_search_quality() — check score thresholds
  • compute_dcg() — rank quality metric
  • merge_dedup_results() — merge two result sets
  • get_ad_images_by_ids() — fetch image URLs from DB
  • get_new_matching_ads() — check for new matches since session start
→ responder_node (fixed)
recommendation_node
When a specific car is NOT available, find similar alternatives using relaxed search criteria.
LLM 1× FallbackLLM.astream() — explain alternatives
Tool Calls
  • embedder.encode() — text → vector embedding
  • qdrant_search.search() — relaxed search (broader year range, same body/budget)
  • verify_results() — filter to active listings
  • get_ad_images_by_ids() — fetch image URLs
→ responder_node (fixed)
advisor_node
Car expert evaluating a specific listing. Answers "is this a good deal?", grounds answers in verified listing data, web search for reliability info.
LLM 2× ainvoke + 1× astream (main response + relevance + preference refine)
Tool Calls
  • build_grounding_block() — create verified listing data block
  • validate_response() — hallucination check on price/year
  • WEB web_search.search() — optional reliability/ common issues search
  • embedder.encode() + qdrant_search.search() — find similar cars
  • upsert_user_preferences() — persist refined preferences
→ responder_node (fixed)
seller_node
Handle seller requests: pricing a car, listing tips, general selling advice. Market price analysis via Qdrant + optional web market data.
LLM 1× ainvoke (extract) + 1× astream (pricing/tips response)
Tool Calls
  • embedder.encode() + qdrant_search.search() — find similar listings for market pricing
  • Price computation: median, min/max, recommended range with km multiplier adjustments
  • WEB web_search.search() — optional market context
  • upsert_user_preferences() — persist seller preferences
→ responder_node (fixed)
guide_node
Answer how-to questions about using the website: posting ads, filtering, saving, comparing, contacting sellers, etc.
LLM 1× ainvoke (topic detect) + 1× astream (guide response)
Tool Calls
  • format_website_guide() — generate website features reference text
→ responder_node (fixed)
general_node
General car knowledge, buying advice, market context, greetings, small talk. Optionally performs web search for time-sensitive questions.
LLM 2× ainvoke (search decision + preference refine) + 1× astream (response)
Tool Calls
  • WEB web_search.search() — conditional, based on LLM decision
  • upsert_user_preferences() — persist extracted preferences
→ responder_node (fixed)
responder_node
Final output node. Streams the accumulated response text as SSE token events and emits structured data events (cars, price_analysis, similar_cars, new_matches).
LLM None (pure orchestration)
Tool Calls
  • SSE event emission (queue-based): tokens, cars, similar_cars, price_analysis, new_match, done
  • insert_chat_message() — save message+response to DB
  • upsert_user_preferences() — save preferences with intent history
→ END (fixed)

4. LLM Fallback Chain

Every LLM call goes through a 3-tier FallbackLLM class. If Groq fails, it falls back to OpenRouter.

      flowchart LR
        REQ[LLM Request] --> GROQ1["Groq (Primary)
        ChatGroq
        openai/gpt-oss-120b"]

        GROQ1 -->|"success"| RES[Response]
        GROQ1 -->|"error / rate limit"| GROQ2["Groq Failover
        ChatGroq
        (backup API key)"]

        GROQ2 -->|"success"| RES
        GROQ2 -->|"error / rate limit"| OR["OpenRouter
        Custom HTTP Client
        nemotron-3-nano-omni
        3-attempt retry
        exp backoff"]

        OR -->|"success"| RES
        OR -->|"all failed"| ERR["Raise Exception"]

6. SSE Event Stream (responder_node → Frontend)

Sequence of events emitted by the responder_node over a single SSE connection during a request.

      sequenceDiagram
        participant U as User
        participant F as Frontend
        participant B as Backend Proxy
        participant C as Chatbot
        participant Q as Qdrant
        participant L as LLM (Groq)

        U->>F: Type message / Speak
        F->>B: POST /api/v1/chat/message
        B->>C: POST /message
        C->>C: Rebuild state from Postgres

        Note over C: preference_extractor → router

        C->>L: classify intent
        L-->>C: intent: search_node

        Note over C: specialist node executes

        alt search_node
          C->>L: build search query
          L-->>C: query + filters
          C->>Q: vector_search()
          Q-->>C: results
          C->>L: generate intro text
          L-->>C: streamed text
        end

        Note over C, F: responder_node streams SSE

        C-->>F: event: status {"Searching..."}
        C-->>F: event: token {"word"}
        C-->>F: event: token {"word"}
        C-->>F: event: cars [...]
        C-->>F: event: new_match [...]
        C-->>F: event: done

        F->>U: Render messages & cars
