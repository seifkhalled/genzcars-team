12 Why Hybrid Search (BM25 + Vector)?

🔍 The Problem: Each approach has blind spots

📄 BM25 Full-Text Only

Good at: Exact keyword matching, rare/technical terms like "BMW 520i 2022"

Misses: Synonyms, paraphrases, semantic meaning — "sedan" won't match "executive saloon"

Score: Keyword overlap only, no understanding of intent

🧠 Vector Embedding Only

Good at: Semantic matching, synonyms, conceptual similarity — "family car" matches "7-seater SUV"

Misses: Exact/tail terms — "BMW 520i 2022" might rank a similar Mercedes higher

Score: Cosine similarity, can drift on rare terms

✅ Hybrid (BM25 + Vector + RRF) — Best of Both
Query → BM25 score + Vector similarity → RRF fusion → Final ranked results
🎯
Exact matches
boosted by BM25
🌊
Semantic matches
caught by vectors
🔀
RRF merges both
no tuning needed
📈
Higher recall
without precision loss

🧪 Real-world example: Searching "luxury German sedan" via BM25 misses "Mercedes E-Class" listings that don't contain those keywords — but vector search catches the semantic match

17 MCP — From REST to Standardized Tools

🔄 Evolution: From REST APIs to MCP

❌ Before — REST APIs

Each graph node had hardcoded HTTP calls to specific backend endpoints:

POST /api/cars/search?brand=BMW
POST /api/cars/price?make=BMW&model=520i
POST /api/cars/compare?id1=X&id2=Y

⚠️ Brittle — endpoint changes break all callers. No schema discovery. No fallback.

→

✅ After — MCP Tools

Nodes call tools through a single registry — server handles routing:

mcp_registry.call_tool("search_cars", {brand: "BMW"})
mcp_registry.call_tool("analyze_market_price", {make: "BMW", model: "520i"})

✅ Standardized — schemas declared on the server, discovered by any caller. In-process or remote, same interface.

🤔 Why MCP Instead of Direct Function Calls?

🔌 Standardized Protocol

Tools, resources, and prompts follow the MCP spec — not custom function signatures. Any MCP-compatible client can discover and use them.

📋 Tool Discoverability

Each server declares metadata (name, description, input schema) so the LLM knows exactly what tools exist and how to call them — no hardcoded tool lists.

🔁 In-Process Transport (Zero Overhead)

Servers run as Python modules directly inside the chatbot process — no subprocess, no network. Stdio/SSE available for future separation if needed.

⚡ Graceful Degradation

Every graph node tries MCP first. If it fails, it falls back to direct Qdrant/PostgreSQL queries. No single point of failure.

21 How the Prompt Builds the Final Output

📝 Prompt Anatomy — 5 Sections Combined
📦 Section 1 — Ad DataRaw DB fields: make, model, year, price, mileage, engine, fuel, transmission, color, location, features
🌐 Section 2 — Web ResearchTavily results: market prices, common issues, expert reviews, owner forums
📷 Section 3 — Computer VisionAI analyzes 10+ car photos per ad: checks visible body condition, paint, interior wear, tire wear, and damage indicators via Vision LLM
📊 Section 4 — Scoring RulesEach category (0-10): Value for Money, Reliability, Running Cost, Resale Value, Condition → weighted Overall
🛡️ Section 5 — Hallucination RulesStrict: trace claims to sources, no unsupported statements, confidence must match data

27 Key Design Decisions

🔎 Hybrid BM25 + Vector Search

Decision: RRF fusion of BM25 full-text + vector cosine search

Alternative considered: Pure vector search with all-MiniLM-L6-v2

Why this won: Pure vector search missed exact keyword matches (e.g., model codes like "520i"). BM25 handles precise term matching while vector captures semantic intent. RRF fusion gives best of both worlds with automatic quality evaluation and broader retry on low scores.

🧠 3-Tier LLM Fallback

Decision: Groq (fast) → Groq (powerful) → OpenRouter → Groq fallback

Alternative considered: Single-provider (Groq only)

Why this won: Groq provides low latency but limited model selection. OpenRouter adds redundancy and access to specialized models (vision, reasoning). The 3-tier chain improves resilience: if one model/provider is unavailable, the app can try the next configured fallback before returning an error.

🛡️ Hallucination Guard Layer

Decision: 5-check validation before returning comparison results

Alternative considered: No validation layer (trust LLM output directly)

Why this won: LLMs confidently fabricate car specifications and pricing. The 5-check system (claim trace-back, accident claim validation, mechanical fault restrictions, confidence consistency, winner validity) catches hallucinations before they reach users, critical for a marketplace where inaccurate data could lead to poor purchase decisions.
