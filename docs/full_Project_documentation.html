0. System Overview

The system consists of three microservices running behind an Nginx reverse proxy, plus a Next.js frontend. Each service is a separate FastAPI application with its own database pool, vector search client, and LLM configuration. A comprehensive monitoring stack (Prometheus + Grafana + Loki) provides observability, and Redis provides distributed caching for expensive computations.

Chatbot Service (:8001) — LangGraph-powered multi-agent chat with MultiLLM routing, hybrid search, semantic caching, guardrails, client disconnect detection, and MCP (Model Context Protocol) tool abstraction layer.
Comparison Analysis (:8002) — Standalone car comparison pipeline with Tavily web research and dual-LLM fallback.
Backend API (:8000) — REST API for ads, auth, images, search, and analytics.
Frontend (:3000) — Next.js app with chat, compare, and listing pages.
Redis (:6379) — Distributed cache for API responses, comparison reports, and LLM results.
Prometheus (:9090) — Metrics collection from all Python services at /metrics.
Grafana (:3001) — Dashboards for metrics (Prometheus) and logs (Loki).
Loki (:3100) — Log aggregation with Promtail collecting container stdout.

1. Chatbot — LangGraph Multi-Agent Flow

Every user message triggers a full graph traversal: preference_extractor → router → specialist → responder. The state is a CarsChatState TypedDict that carries messages, preferences, retrieved ads, and routing info across nodes. After the responder emits all SSE events, the state's messages list is updated with the new AI response and persisted to PostgreSQL.

2. Node Details & Tool Calling

preference_extractor every turn

→ {"preferences": merged, "turn_count": +1}
LLM 1× MultiLLM.ainvoke_task("preference_extractor", ...) — routed to fast model (Groq, low temp, 1024 tokens)

  • upsert_user_preferences() — persist to PostgreSQL (fire-and-forget)
→ router (fixed edge)
router every turn
Intent classifier — decides which specialist node should handle the request. Uses conversation history, context_ad_id, and current preferences. Returns one of 5 node names as a string. Responses are cached via SemanticCache for identical prompts (2min TTL).
→ {"next_node": str, "intent": str, "intent_history": [...]}

Routing Rules

  • advisor_node: specific car in context, "is this a good deal?", "should I buy?"
  • seller_node: "I want to sell", "price my car", "listing advice"
  • guide_node: website help, how to post/filter/compare
  • general_node: car knowledge, greetings, unclear requests
→ one of 5 specialist nodes (conditional via next_node state key)
catalogue_node
Determines if the request is for a specific known car (brand+model+year) or a broad search. If specific, checks PostgreSQL catalogue for exact matches via MCP. Routes to search_node if available, recommendation_node if not.

→ search_node (available or broad) | recommendation_node (not available)
search_node
Hybrid search (BM25 keyword + vector semantic) against Qdrant via MCP. Builds an optimized query via LLM, enriches with preference filters, performs initial search via MCP search_cars tool with quality evaluation. If quality is poor, retries with broader query and merges results. Embeddings cached via SemanticCache (2h TTL). Also checks for new listings since session start.

Car expert evaluating a specific listing. General car knowledge, buying advice, market context, greetings, small talk. Optionally performs DuckDuckGo web search for time-sensitive questions (reliability, prices, reviews, news). Uses an LLM sub-call to decide whether to search (search decision cached).

4. MCP (Model Context Protocol) Architecture

The MCPToolRegistry acts as the single interface for all graph nodes to call external tools. Two in-process MCP servers (car-search and analysis) provide 10 tools total, wrapping Qdrant hybrid search, PostgreSQL queries, price analysis, web search, brand expansion, and website guide generation.

MCPToolRegistry — defined in app/core/mcp_client.py (152 lines).

Supports three transport modes: in-process (same process, zero overhead), stdio (subprocess for isolation), and SSE (remote HTTP servers). Currently all servers run in-process.

5. Multi-LLM Routing & Fallback Chain

Every LLM call goes through the MultiLLM class which provides task-based routing — simple tasks (routing, extraction, topic detection) use a cheap/fast Groq model (low temp, 1024 tokens), while complex tasks (advisor, seller, search) use a powerful model (Groq fallback → OpenRouter → fast fallback, 4096 tokens, streaming). This reduces cost and latency for routine operations while preserving quality for reasoning-heavy nodes.

6. Caching, Memory & Persistence

The system uses a tiered caching strategy: in-memory caches for hot data within the chatbot process, and Redis as a distributed cache shared across all services. Conversation memory uses PostgreSQL for long-term persistence and MemorySaver (in-memory) for LangGraph checkpoints during active sessions. A lightweight SemanticCache (in-memory LRU) reduces redundant LLM and embedding calls. Redis caches expensive API responses and comparison reports.

Redis Cache (Distributed)

Redis runs as a Docker container (cars_redis) on port 6379 and is accessible by all services on the cars_net Docker network. Configuration via REDIS_URL=redis://redis:6379/0 in .env. The Backend config already defines redis_url: str | None = None at Backend/app/config.py:34, and the Comparison Analysis config has redis_url: str = "redis://redis:6379/0".

  4. Session restore: frontend calls GET /history/{session_token} on mount to reload past messages from DB.

Session Token

Generated by POST /chat/session, stored in localStorage on the frontend, and sent with every /message request. Persists across page refreshes and browser restarts.

7. Guardrails & Input Safety

A lightweight guardrails layer runs at the entry point of every chat request, before any LLM processing. It validates input, detects prompt injection, and provides fallback error messages to the user without invoking the graph.

guardrails.validate_input() — called in routers/chat.py before graph execution.
Checks: empty message, max length (2000 chars), prompt injection patterns, sensitive data detection.
On failure: returns {"type": "error", "content": "..."} and done SSE events immediately.

Injection Pattern Detection

The guardrail blocks attempts to override system instructions via regex patterns matching:

  • "ignore all previous instructions"
  • "you are now free / unleashed / DAN"
  • "new instructions:"
  • "you don't have to obey"

Tavily Web Research in Detail

The TavilyWrapper.research_car(ad) method in app/core/tavily.py performs 3 parallel web searches per car, each configured with search_depth="basic", max_results=3, and include_answer=True (Tavily returns an AI-generated summary alongside raw results):

Search Query Result Field Purpose
"{brand} {model} {year} reliability common problems Egypt" reliability_research Find known mechanical issues, common problems, reliability ratings, and recalls specific to the Egyptian market.
"{brand} {model} {year} price Egypt market 2024 2025" price_research Find current market pricing, resale value trends, and price ranges in Egypt.
"{brand} {model} owner review Egypt مصر" reputation_research Find real owner reviews, ownership experiences, and word-of-mouth reputation. The Arabic keyword مصر ensures Egypt-specific results.

Each Tavily result contains a tavily_answer (LLM-generated summary) and sources[] (up to 3 articles with title, URL, and content snippet). These results are then injected verbatim into the LLM prompt in the ANALYZE stage, giving the AI current, real-world context beyond what the ad listing provides. The research_sources in the final report are deduplicated URLs across all 3 research categories per car.

Car Image Analysis in the ANALYZE Stage

The Comparison Analysis service analyzes one image per car — the cover image (cover_image_url, which is the ad image with order_index = 0). This is fetched from PostgreSQL via the LEFT JOIN ad_images query in fetch_ads_for_comparison().

How the image is used

In analyzer.py, the _parse_llm_json() function calls _build_multimodal_content(human_text, cover_image_url) which constructs the message as a list of content blocks: 

[
  {"type": "text", "text": "Analyze this car: BMW 320i 2020..."},
  {"type": "image_url", "image_url": {"url": "https://...cover.jpg"}},
]

This multimodal message is sent to OpenRouter with the model google/gemini-2.0-flash-exp:free (which supports vision). The prompt instructs the LLM to produce a photo_analysis field:

"2-3 sentences analyzing the car photos: exterior condition, visible damage (scratches, dents, rust), tire wear, cleanliness, any visual red flags. Base this on the image provided above."

If the OpenRouter (multimodal) call fails, the pipeline falls back to Groq, which does NOT support vision — so the fallback analysis is text-only with no image available.

Backend Comparison — Vision Analysis (separate service)

The Backend API also has a comparison endpoint at POST /api/v1/compare-ai (powered by ComparisonService in Backend/app/services/comparison/). Unlike the standalone microservice, this endpoint:

  • Compares exactly 2 ads (no 3-ad mode)
  • Runs no Tavily web research — pure data-driven analysis
  • Uses a dedicated vision analyzer that collects up to 5 images per car (cover + all additional images)
  • Has a hallucination guard that validates claims against source data

Vision analysis flow (Backend)

The vision_analyzer.py collects all available images (cover first, then up to 4 additional from ad_images) and sends them to OpenRouter (google/gemini-2.0-flash-exp:free) with a structured inspection prompt that examines 5 areas:

Area What the LLM Examines
Exterior Paint condition, scratches, dents, rust, panel gaps, accident indicators
Wheels & Tires Condition, curb rash, tire tread depth
Lights Headlights, tail lights for cracks or fogging
Interior Seat condition, dashboard wear, steering wheel wear, cleanliness
Glass Windshield cracks or chips

The vision analyzer returns structured findings per area with severity (good/fair/poor) and confidence (high/medium/low), plus an overall_condition, accident_indicators (yes/no/unknown), and a computed confidence score (0.0–1.0). The vision confidence is then multiplied with the structured analysis confidence to produce the final per-car confidence score: 

combined_confidence = structured.confidence × vision.confidence

This confidence is used by the hallucination guard to prevent the comparison from making claims that exceed what the data supports (e.g., asserting accident history when the vision analyzer found no indicator).

LLM Configuration

The Comparison Analysis uses its own LLM stack, separate from the Chatbot:

  • Primary: OpenRouterChat — custom BaseChatModel subclass with 120s timeout, 3-attempt retry with exponential backoff, model google/gemini-2.0-flash-exp:free (supports vision).
  • Fallback: ChatGroq — model openai/gpt-oss-120b, key from groq_api_key (text-only, no vision).
  • Web search: TavilyWrapper — search_depth="basic", 3 results per query, 3 queries per car.

12. AI Pricing — Fixes for Missing Numbers

The AI pricing system produces price numbers (EGP ranges, median, min/max) for users asking "how much is my car worth?" or "price my car". Two services generate these prices: the Chatbot seller_node (uses Qdrant listing data) and the Comparison Analysis price_analysis endpoint (uses web search + LLM knowledge). Below are the issues found and the fixes applied to ensure prices are always returned — never zero or missing.

Problem 1: DuckDuckGo Fails → Returns Zero Prices

File: Comparison_Analysis/app/routers/price_analysis.py

DuckDuckGo is rate-limited for automated queries. When it returned empty results, the old prompt told the LLM "set all values to 0" and the _fallback_estimate() function always returned {low: 0, high: 0, average: 0} — making the frontend display no prices.

Fix 1a — Tavily Fallback Search

When DuckDuckGo returns no snippets, the endpoint now tries Tavily (which was already initialized in app.state.tavily but never used in price analysis). If Tavily finds results, those snippets are passed to the LLM prompt instead of "No search results found.":

        if tavily_result and tavily_result.get("results"):
            # Format Tavily results as snippets for the LLM prompt
            ...

When both DuckDuckGo and Tavily fail, the new _fallback_estimate function asks the LLM directly for a price estimate based on its training knowledge instead of returning zeros: 

async def _fallback_estimate(llm, make, model, year):
    if llm:

        # Returns { "estimated_range": { "low", "high", "average" } }
        ...
    # Only if LLM also fails: return 0/0/0 with summary

Fix 1c — Prompt Change: No More "Set All to 0"

The PRICE_SYSTEM and PRICE_HUMAN_TEMPLATE prompts were changed from "set all values to 0" to "use your knowledge of the Egyptian car market to provide a reasonable estimate", allowing the LLM to produce prices even without web search hits.

Problem 2: No Qdrant Listings → Generic Tips Instead of Prices

File: Chatbot/app/graph/nodes/seller_node.py

When the user asks "how much is my BMW 2020 worth?" but no comparable listings exist in Qdrant, price_analysis was None, and the node fell back to TIPS_SYSTEM — which gives photography/listing advice without any price numbers. Very frustrating for users asking for a price.

Fix 2 — Web-to-Price Prompt Fallback

A new PRICING_WEB_SYSTEM prompt was added. The logic now chooses between three paths instead of two:

    if price_analysis:
        # Path A: Qdrant data exists → PRICING_SYSTEM (hard numbers)
    elif web_context:
        # Path B: No Qdrant, but web search found context → PRICING_WEB_SYSTEM
    else:
        # Path C: Nothing → TIPS_SYSTEM (listing advice only)

The new PRICING_WEB_SYSTEM prompt tells the LLM: "use your expertise to estimate a reasonable price range for the Egyptian market. Be specific — provide an estimated price range in EGP." This ensures users always get a price estimate even without Qdrant data.

Combined Impact

With all three fixes, the AI pricing pipeline now has 4 layers of fallback before returning no data:

  1. Qdrant listing data (most accurate — from active marketplace listings)
  2. DuckDuckGo web search (current market prices from Egyptian car sites)
  3. Tavily web search (AI-summarized market research as fallback)
  4. LLM knowledge (the AI's training data about Egyptian car prices)

Only if all four layers fail does the system return 0/"No market data available".

Detailed Implementation

STT — Audio Transcription (Chatbot/app/voice/stt.py)

  • Endpoint: Groq API POST /openai/v1/audio/transcriptions with model whisper-large-v3-turbo.
  • Key fallback logic: The function builds a list of 1–3 Groq API keys (groq_api_key, groq_api_key_fallback, groq_api_key_fallback2) and tries each sequentially. If all fail, raises RuntimeError("STT service returned {code}").
  • File format handling: Extracts file extension from filename. If unsupported or missing, defaults to webm. Allowed: webm, wav, mp3, m4a, ogg, flac.
  • Size limit: 25MB maximum file size (MAX_FILE_SIZE = 25 * 1024 * 1024). Raises ValueError if exceeded.
  • Confidence: Extracts per-segment confidence scores from Groq's verbose_json response and computes an average. Returns None if no segments have confidence data.
  • Language detection: Groq returns the detected language code. If it's not "ar" or "en", defaults to "en".

TTS — Speech Synthesis (Chatbot/app/voice/tts.py)

  • Engine: edge-tts (Microsoft Edge Text-to-Speech API via the edge-tts Python package).
  • Voice mapping: {"ar": "ar-EG-SalmaNeural", "en": "en-US-JennyNeural"}. Unknown languages default to en-US-JennyNeural.
  • Streaming: Uses edge_tts.Communicate(text, voice).stream() which yields chunks of type {"type": "audio", "data": bytes}. All audio chunks are concatenated into a single bytes object.
  • Max text length: 2000 characters (MAX_TEXT_LENGTH = 2000). Text is silently truncated before synthesis.
  • Output: Returns audio/mpeg bytes (MP3 format).

Frontend Recording Logic (Frontend/hooks/useVoice.ts)

  • Browser support check: Verifies navigator.mediaDevices?.getUserMedia and MediaRecorder are available.
  • MIME type selection: Prefers audio/webm, falls back to audio/mp4.
  • Recording: startRecording() gets microphone access, creates a MediaRecorder, collects chunks via ondataavailable.
  • Upload: stopRecording() stops the recorder, stops all media tracks, builds a Blob, appends it to a FormData as recording.webm, and POSTs to {API_URL}/chat/stt. Resolves with { text, language }.
  • Error handling: If HTTP response is not OK, throws error with response body. If microphone access is denied, sets state to "error" and re-throws.

Frontend TTS Logic (Frontend/hooks/useTTS.ts)

  • Playback: speak(text, language) stops any current playback first, fetches MP3 from /chat/tts, creates an ObjectURL, creates an Audio element, and plays it.
  • Lifecycle: Tracks state: idle → loading → playing → idle (on end) or error (on failure). Returns a Promise that resolves when playback ends.
  • Cleanup: stop() pauses the audio and calls URL.revokeObjectURL() to free the blob URL. Also called automatically when playback ends naturally.

Backend Proxy Routes (Backend/app/routers/chat.py)

RouteMethodInputForwarded ToTimeout
